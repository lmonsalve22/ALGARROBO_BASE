## ARCH-C2-001 — Dualidad de cargadores de datos: `DataManager` vs `dataLoaderCead`

**🏗️ Aspecto**: Capa de Datos — Dualidad de implementaciones coexistentes

**📐 Patrón Detectado**: Singleton (DataManager) + Object Literal (dataLoaderCead) en paralelo

**⚠️ Riesgo Arquitectónico**: `data_cead.js` define `dataLoaderCead` con su propia lógica de fetch, decompress y estado (`STATE_DATA_CEAD`), **desconectada** del `DataManager` centralizado. Aunque la carga del loader está comentada (línea 385-389: `DISABLED: Using Unified DataManager`), el archivo sigue existiendo con lógica activa. Cualquier desarrollador podría invocarlo por error, causando doble carga y race conditions de estado.

**📈 Recomendación**: 
1. Convertir `data_cead.js` en un archivo de **solo configuración** (COLS_CEAD + STATE_DATA inicial).
2. Mover toda la lógica de `dataLoaderCead.load()` al `DataManager.loadCeadData()` (ya implementado).
3. Eliminar el objeto `dataLoaderCead` para prevenir confusiones.

## ARCH-C2-002 — `pdf.js` acoplado a `window.STATE_DATA` y `window.IAModule` directamente

**🏗️ Aspecto**: Inversión de Dependencias — PDFModule como consumidor de globals

**📐 Patrón Detectado**: God Object consumer / Tight Coupling

**⚠️ Riesgo Arquitectónico**: `PDFModule` accede a `window.STATE_DATA`, `window.STATE_DATA_CEAD`, `window.IAModule`, `window.App` y `window.jspdf` directamente. Si cualquiera de estos globals no está disponible en el momento de ejecución, el módulo falla silenciosamente con datos incompletos (ej. `state.comunaName` = undefined). Esto hace la sequencia de carga extremadamente frágil ante cambios de orden de scripts.

**📈 Recomendación**: Inyectar dependencias como parámetros en `exportReport(dataContext, iaContext)` en lugar de leerlas del scope global en cada método. Permite testing unitario y desacoplamiento.

```javascript
// ANTES: acoplado
generateAISummaryText() {
    const state = window.STATE_DATA || {};
    if (typeof IAModule !== 'undefined' && IAModule.interpretations?.vista1) { ... }
}

// DESPUÉS: desacoplado
generateAISummaryText(state, interpretations) {
    if (interpretations?.vista1) { ... }
}
```

## ARCH-C2-003 — `RID.View` y `RID.UI` como namespace fragmentado sin cohesión

**🏗️ Aspecto**: Module Boundary — Namespace RID incompleto

**📐 Patrón Detectado**: Namespace parcial (God Namespace anti-pattern potencial)

**⚠️ Riesgo Arquitectónico**: El namespace `RID` fue creado en `rid-core.js` para centralizar módulos, pero contiene solo placeholders (`RID.State`, `RID.DataManager`, `RID.IA`, `RID.UI`). En la práctica, `RID.View` (view-controller) y `RID.UI` (ui-helper) están implementados, pero el resto del sistema ignora `RID.State` y sigue usando `window.STATE_DATA` directamente. El namespace es un promesa no cumplida que añade confusión sin beneficio actual.

**📈 Recomendación**: 
- **Corto plazo**: Eliminar `rid-core.js` y los placeholders vacíos.  
- **Largo plazo**: Implementar `RID.State` como store reactivo (Proxy) y migrar `STATE_DATA` gradualmente.

## ARCH-C2-004 — `PDFModule` y `App` ambos gestionan export: responsabilidad solapada

**🏗️ Aspecto**: SRP — Separación de Responsabilidades entre App y PDFModule

**📐 Patrón Detectado**: Feature Envy / Responsibility Overlap

**⚠️ Riesgo Arquitectónico**: `App.exportPdf()` (app.js) gestiona el export simple (5 vistas de seguridad), mientras que `PDFModule.exportReport()` (pdf.js) gestiona el export completo (45 vistas con portada, TOC, contraportada). Ambos usan html2canvas + jsPDF, crean overlays propios, gestionan `Chart.defaults.animation`. Hay lógica duplicada de captura y overlay. El desarrollador debe saber cuándo usar cuál.

**📈 Recomendación**: Unificar bajo `PDFModule`. `App.exportPdf()` debería delegarse a `PDFModule.exportSimple(views)` para eliminar la duplicación y centralizar el comportamiento de export.

## ARCH-C2-005 — Patrón de Comunicación: eventos Custom vs polling coexisten

**🏗️ Aspecto**: Communication Pattern — Inconsistencia entre event-driven y polling

**📐 Patrón Detectado**: Hybrid (Custom Events + setTimeout polling)

**⚠️ Riesgo Arquitectónico**: Tras el Ciclo 1, `ia2.js` usa eventos (`dataManagerLoaded`, `viewLoaded`) correctamente. Sin embargo, `view-controller.js` sigue usando polling con `setTimeout(check, 100)`. Tener dos patrones de sincronización paralelos aumenta la carga cognitiva y hace difícil razonar sobre el orden de inicialización del sistema.

**📈 Recomendación**: Migrar `RID.View.init` a escuchar `dataManagerLoaded` si `STATE_DATA.isLoaded` ya es false, o ejecutar inmediatamente si ya es true. Eliminar el polling:

```javascript
init(callback, options = {}) {
    const source = options.source || 'stop';
    const key = source === 'cead' ? 'STATE_DATA_CEAD' : 'STATE_DATA';

    const execute = () => {
        const S = window[key];
        const C = source === 'cead' ? window.COLS_CEAD : window.COLS;
        callback(S, C).catch(err => LOG.error('RID.View.init error:', err));
    };

    const target = window[key];
    if (target?.isLoaded) {
        // Ya listo: ejecutar en el próximo tick (no bloquear parsing)
        queueMicrotask(execute);
    } else {
        // Esperar evento
        const eventName = source === 'cead' ? 'dataCeadLoaded' : 'dataManagerLoaded';
        window.addEventListener(eventName, execute, { once: true });
    }
}
```

**📈 Beneficio**: Elimina polling, reduce latencia de inicialización de vistas, unifica el patrón de comunicación.
