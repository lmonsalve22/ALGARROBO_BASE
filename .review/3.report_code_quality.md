## CQ-C2-001 — Dead Code: `data_cead.js` bloque `if/else` para COLS_CEAD

**💡 Hallazgo**: Las líneas 31-50 de `data_cead.js` definen una configuración fallback de `COLS_CEAD` y loguean "Using Centralized COLS_CEAD", pero en línea 51 el objeto se redefine incondicionalmente. El bloque `if/else` nunca tiene efecto. Es Dead Code garantizado.

**🏗️ Principio/Métrica**: DRY / Dead Code

**📍 Ubicación**: `data_cead.js` líneas 31-50

**🛠️ Refactor Sugerido**: Eliminar el bloque completo. Dejar solo la definición canónica desde línea 51.

**📈 Beneficio**: -20 líneas, eliminación de false-positive en logs de producción.

## CQ-C2-002 — Layout Thrashing latente en `pdf.js`: read/write en loop

**💡 Hallazgo**: `PDFModule.generateTableOfContents()` (línea 218) llama a `pdf.getTextWidth(truncatedTitle)` dentro del `forEach` de secciones. Aunque `getTextWidth` es una operación sobre el canvas de jsPDF (no el DOM), el problema real está en `updateProgress()` (líneas 737-754) que consulta 5 `getElementById` y luego escribe en sus propiedades dentro de cada vuelta del loop de exportación.

**🏗️ Principio/Métrica**: DOM Access / Layout Thrashing (INP)

**📍 Ubicación**: `pdf.js` → `updateProgress()` líneas 737-754

**🛠️ Refactor Sugerido**: Cachear referencias DOM en el `showEnhancedOverlay()` donde se crean los elementos:
```javascript
// En showEnhancedOverlay, tras appendChild:
this._domRefs = {
    progressBar: document.getElementById('pdfProgressBar'),
    progressPercent: document.getElementById('pdfProgressPercent'),
    progressLabel: document.getElementById('pdfProgressLabel'),
    currentStep: document.getElementById('pdfCurrentStep'),
    phaseText: document.getElementById('pdfPhaseText')
};

// En updateProgress(), usar this._domRefs.progressBar directamente
```

**📈 Beneficio**: Elimina 5 queries DOM por iteración de vista (hasta 45+ queries en un export completo).

## CQ-C2-003 — Naming no semántico: `dataLoaderCead` como objeto suelto en `data_cead.js`

**💡 Hallazgo**: `dataLoaderCead` es un objeto con comportamiento de servicio pero con naming de variable local. Su método `load()` carga y muta el global `STATE_DATA_CEAD` sin encapsulamiento. Es funcionalmente un "God Side-Effect Object".

**🏗️ Principio/Métrica**: Naming / SRP (Single Responsibility)

**📍 Ubicación**: `data_cead.js` línea 280

**🛠️ Refactor Sugerido**: Renombrar a `CeadDataService` y encapsular la mutación de estado dentro del objeto:
```javascript
// ANTES: muta window.STATE_DATA_CEAD directamente
STATE_DATA_CEAD.isLoaded = true;

// DESPUÉS: método encapsulado
CeadDataService._setState(data) { ... }
```

**📈 Beneficio**: Mantenibilidad. Preparación para futura migración a módulo ES.

## CQ-C2-004 — `pdf.js`: Patrón `// fase 1`, `// fase 2` sin abstracción de función

**💡 Hallazgo**: `exportReport()` (visible a partir de línea ~800) ejecuta múltiples fases (portada, TOC, vistas, contraportada) como bloque secuencial monolítico. No hay funciones extraídas por fase, lo que hace el mantenimiento de orden difícil y el testing imposible.

**🏗️ Principio/Métrica**: DRY / SRP

**📍 Ubicación**: `pdf.js` → `exportReport()` 

**🛠️ Refactor Sugerido**: Extraer helpers:
```javascript
async _capturePhase(pdf, views) { ... }   // P1: captura de vistas
async _buildCoverPhase(pdf) { ... }       // P0: portada
async _buildTOCPhase(pdf, sections) { ... }// TOC
```

**📈 Beneficio**: Legibilidad, testabilidad unitaria por fase.

## CQ-C2-005 — `ui-helper.js`: `UIHelper.setHtml()` sin alternativa segura documentada

**💡 Hallazgo**: `UIHelper.setHtml(id, html)` asigna `innerHTML` directamente. No existe documentación de que sólo debe usarse con HTML generado internamente. Las vistas pueden llamarla con datos del servidor por error.

**🏗️ Principio/Métrica**: Defensive Programming / Safety

**📍 Ubicación**: `utils/ui-helper.js` → línea 70

**🛠️ Refactor Sugerido**:
```javascript
// Añadir versión segura por defecto
setHtmlSafe(id, text) {
    const el = document.getElementById(id);
    if (el) el.textContent = text; // usa textContent, no innerHTML
},
// Renombrar setHtml → setHtmlTrusted con JSDoc @internal
```

**📈 Beneficio**: Prevención de XSS accidental por llamadores que confunden métodos.

## CQ-C2-006 — `view-controller.js`: `console.error` directo (inconsistencia de logger)

**💡 Hallazgo**: `view-controller.js` usa `console.error` y `console.log` directamente (líneas 58 y 64), cuando el resto del proyecto usa el `LOG` condicional de `logger.js`. Esto divulga stack traces de errores en producción.

**🏗️ Principio/Métrica**: Consistency / Information Disclosure

**📍 Ubicación**: `utils/view-controller.js` → líneas 58, 64

**🛠️ Refactor Sugerido**:
```javascript
// Reemplazar:
console.error('❌ RID.View.init error:', err);
console.log('✅ RID.View Controller loaded.');

// Por:
LOG.error('❌ RID.View.init error:', err);
LOG.info('✅ RID.View Controller loaded.');
```

**📈 Beneficio**: Consistencia con política de logging del sistema. Supresión en producción.
